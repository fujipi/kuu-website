AIエージェントをステージングで合格させ、本番にリリースしたら評価は終わり——そう運用しているチームでは、静かな品質劣化に気づかないまま数週間が過ぎることがある。ユーザーの入力分布は変化し、外部ツールの API レスポンスは揺れ、モデルバージョンは更新される。オフラインのゴールデンデータセットはこれらの変化を映し出さない。

## オフライン評価はなぜ本番の品質保証に不十分か

> オフラインテストセットは固定された分布のスナップショットに過ぎず、本番で起きる分布シフト・新種エッジケース・スケール特有の挙動を事前に網羅することは不可能だ。

本番環境がオフライン評価と乖離する原因は主に3つある。**分布シフト**（ユーザー行動の変化でリクエストパターンが変わる）、**ツール依存性**（外部 API のレスポンス変化がエージェントの推論に波及する）、**スケール特有の挙動**（少量テストでは現れなかったエラーが高トラフィックで顕在化する）だ。

1,200本の本番デプロイメントを分析した ZenML の調査（2025年）によれば、品質問題が顕在化するのはデプロイ後の運用期間中が最も多く、事前評価だけでは捕捉できないことが示されている。[可観測性・トレース設計](/blog/agent-observability-tracing-instrumentation/)と並行して、オンライン評価パイプラインを最初から組み込む必要がある。

## 3層サンプリング評価アーキテクチャとは何か

> オンライン評価は「ヒューリスティック全量・LLMジャッジ5〜10%・人間アノテーション2〜5%」の3層で設計する。コストと精度のトレードオフを層ごとに役割分担することが重要だ。

### 第1層: ヒューリスティック評価（全量・リアルタイム）

レイテンシ、エラー率、トークン消費量、出力フォーマットの整合性など、ルールベースで即座に判定できる指標を全リクエストに適用する。判定はミリ秒単位で完了し、コストはほぼゼロだ。スコアが閾値を超えたトレースは自動的に第2層へフラグを立てる。

### 第2層: LLM-as-a-judge 評価（5〜10%サンプリング）

フロンティアモデルをジャッジとして使用し、回答の正確性・指示追従・推論品質を採点する。全量適用はコストが高すぎるため、統計的に有意なサンプル（5〜10%）に絞るのが実践的な設計だ。CI/CD の回帰ケースと第1層でフラグされたトレースには100%適用する。[LLM-as-a-judge の採点設計](/blog/llm-as-a-judge-agent-evaluation-enterprise/)と組み合わせることでルーブリックの精度を高められる。

### 第3層: 人間アノテーション（2〜5%・戦略的サンプリング）

自動評価ジャッジのキャリブレーション、未知のタスクタイプ、第2層で高不確実性と判定されたケースに人間レビューを充てる。全量に人手を掛けることはスケールしないが、2〜5%の戦略的サンプリングにより自動評価の精度を継続的に担保できる。

## シャドーテストはどう実装するか

> シャドーテストは本番リクエストを候補モデルへ複製送信し、ユーザーに見えない形でレスポンスを比較する手法だ。API Gateway かロードバランサーで複製し、候補のレスポンスはログに格納するだけで返却しない。

シャドーテストの実装手順は次の通りだ。

1. **トラフィック複製**: API Gateway またはアプリケーション層で、本番モデルと並行して候補モデルへも同一リクエストを非同期送信する。ユーザーには本番モデルの応答のみを返すため、体験への影響はない。
2. **レスポンスキャプチャ**: 両モデルの出力・レイテンシ・トークン数をメタデータとともに構造化ログに記録する。本番データの暗号化・データ保持ポリシーを候補側にも同様に適用する。
3. **比較分析**: オフラインでベースラインと候補のレスポンスを第2層の LLM ジャッジで採点し、品質スコア・コスト・レイテンシを比較する。

期間は「最低1ビジネスサイクル」を確保する。週次・月次パターンを含むことで、ピークトラフィックとエッジケースを含む代表的な分布を確保できる。

シャドーテストで明確な回帰がなければ A/B テストへ移行する。A/B テストではユーザーの一部（カナリア比率、例えば5%）を新バリアントに切り替え、タスク完了率・ユーザー満足度など主観的な品質シグナルを計測する。シャドーテストが捕捉できない「ユーザー体験」の変化をここで検証する。

## 本番ドリフト検知と回帰防止をどう設計するか

> 品質劣化は急激に起きるとは限らない。入力分布の時系列変化を埋め込みベクトル空間でモニタリングし、エッジケースを自動的に回帰テストセットへ追加することで劣化を早期に捉え続ける。

ドリフト検知の実装要素は3つだ。**時系列分析**（スコアの移動平均・パーセンタイルをトラック別に計測し統計的有意な下降を検知）、**埋め込み異常検知**（入力の埋め込み分布が訓練データから大きく外れたリクエストをフラグ）、**コホートセグメンテーション**（ユーザー属性や入力タイプ別にスコアを分割し、特定セグメントのみで起きる劣化を可視化）だ。

検知されたエッジケースは自動的に回帰データセットへ追加する。[ゴールデンデータセットの設計](/blog/agent-regression-test-golden-dataset/)と連動させることで、オフライン評価の分布が本番に追いついていく。「本番で見つかったバグはテストセットに必ず戻す」サイクルを機械的に回すことが継続評価の核心だ。

大規模なエンタープライズ環境でこの評価パイプラインを構築する場合、[LLM ゲートウェイ](/blog/llm-gateway-routing-rate-limiting/)と評価レイヤーの統合が鍵になる。Kuu の [RDE（Reinvention Deployed Engineering）](/services/rde/) では、評価パイプラインの設計から本番モニタリングの基盤整備まで一貫して支援する。

## まとめ

本番 AI エージェントの品質保証はオフライン評価だけでは完結しない。ヒューリスティック全量・LLM ジャッジ 5〜10%・人間アノテーション 2〜5% の3層サンプリングでコストと精度を配分し、シャドーテストで安全に候補モデルを検証し、ドリフト検知で継続的な品質劣化を捉える——この三段構えが本番エージェントの評価基盤となる。エンタープライズ規模での評価体制の設計・実装については、[Kuu の RDE サービス](/services/rde/)からご相談ください。
