オフライン評価でゴールデンデータセットを通過したエージェントも、本番のユーザー入力分布が変化したり、上流ツールの挙動が変わったりすることで品質劣化が生じる。問題が顕在化するのは、ユーザーエスカレーションや業務停止が発生してからというケースが多い。オンライン評価は本番トラフィックをサンプリングしてリアルタイムで品質指標を取得し、劣化を早期に検知するフィードバックループを構築する設計だ。

オフライン評価の設計（ゴールデンデータセット・回帰テスト）については[エージェント回帰テストとゴールデンデータセット](/blog/agent-regression-test-golden-dataset/)、トレース計装については[AIエージェントの可観測性](/blog/agent-observability-tracing-instrumentation/)を先に参照してほしい。

## オンライン評価とはどういう仕組みか

> オンライン評価は本番トラフィックのサブセットに評価ジョブを非同期で実行し、オフライン評価では見えない本番固有の品質劣化を継続的に検知する仕組みだ。

オフライン評価は「決められたテストセット」への出力を採点する静的な評価だ。これに対してオンライン評価は、実際のユーザーリクエストを流れるままに計測する動的な評価だ。両者は補完関係にある。

| 観点 | オフライン評価 | オンライン評価 |
|---|---|---|
| 計測対象 | 固定テストセット | 本番トラフィック（サンプリング） |
| 主な用途 | デプロイ前ゲート・回帰防止 | 劣化検知・分布シフト検知 |
| タイミング | CI/CDパイプライン | 継続的・非同期 |
| コスト | 予測可能 | サンプリング率で制御 |

2026年時点での主流設計は「OpenTelemetry GenAI Semantic Conventions準拠のトレースデータを評価パイプラインに接続し、品質スコアをカスタムメトリクスとしてAPM基盤に戻す」アーキテクチャだ。Langfuse・LangSmith・Arize Phoenixがこのパターンを実装している。

エンタープライズのLLMエージェント評価基盤の構築については[Kuuの RDEサービス](/services/rde/)で支援している。

## サンプリング戦略をどう設計するか

> サンプリングはヘッドベース・テールベース・シャドートラフィックの3種類があり、目的とコストに応じて使い分ける。本番では3方式の組み合わせが最も実効性が高い。

**ヘッドベースサンプリング**

トレース開始時にサンプリング対象を決定する。実装がシンプルで、OpenTelemetryのデフォルト動作に近い。本番負荷が高いシステムで均一にサンプリングしたい場合に適している。一般的な設定は本番トラフィックの10〜20%だ。

```python
from opentelemetry.sdk.trace.sampling import TraceIdRatioBased

# 本番トラフィックの10%をサンプリング
sampler = TraceIdRatioBased(0.1)
```

**テールベースサンプリング**

スパン完了後に保持・廃棄を判断する。エラーや低品質スコアが付いたトレースを優先的に保持できる。問題ケースの根本原因分析に特に有効だが、コレクター側のメモリバッファが必要だ。

```python
# テールベース: エラートレースは必ず保持、正常は10%
if trace.has_error or trace.quality_score < THRESHOLD:
    keep = True
else:
    keep = random.random() < 0.10
```

**シャドートラフィック（シャドーモード評価）**

新バージョンのエージェントに本番トラフィックのコピーを流し、本番応答には影響を与えずに品質を比較する。モデルアップグレードやプロンプト変更のリスクを測定するために使う。

| 方式 | 適用場面 | 注意点 |
|---|---|---|
| ヘッドベース | 均一サンプリング・ベースライン取得 | 低頻度の異常を見逃しやすい |
| テールベース | 異常トレースの保全・根本原因分析 | コレクター側のバッファコストが高い |
| シャドートラフィック | モデル/プロンプトアップグレード前検証 | 2倍の推論コストが発生する |

## LLM-as-judgeのコスト最適化をどう設計するか

> LLM-as-judgeは高精度だが高コストのため、本番の5〜10%に絞り、残り90%以上にはヒューリスティック評価とEmbedding類似度を組み合わせるのが推奨パターンだ。

**2層評価アーキテクチャ**:

```
100% トラフィック
    └── 高速ヒューリスティック評価（ルールベース）
          └── 異常フラグ付きトレース（5〜10%）
                └── LLM-as-judge（詳細採点）
```

高速ヒューリスティック評価の例:
- レスポンス長・トークン数の外れ値検出
- ツール呼び出し失敗率・リトライ率
- 出力テキストの禁止フレーズ検出
- Embedding類似度を使った回答妥当性スコア

LLM-as-judgeのサンプリング対象はヒューリスティックがフラグを立てたトレースに限定する。これにより評価コストを大幅に削減しながら、実際に品質劣化したケースを高確率でカバーできる。採点結果はOpenTelemetryのスパン属性（`eval.quality_score`等）として書き戻し、レイテンシーやトークンコストと同一ダッシュボードで確認できるようにする。

## ドリフト検知とアラート設計の実装パターン

> 品質スコアを時系列で管理し、7日移動平均比5%以上の低下が続く場合にアラートを発火させる設計が本番システムで広く採用されている。

**アラート設計の2段階構成**:

1. **ベースライン確立**: 新バージョンデプロイ後48〜72時間でスコアのベースラインを計測する
2. **閾値の2段設計**: WARNING（5%低下）とCRITICAL（15%低下）の2段階で発火させ、CRITICAL時は自動ロールバックトリガーと連動する

```yaml
# アラート例（Prometheus/Alertmanager形式）
- alert: AgentQualityDegradation
  expr: |
    avg_over_time(agent_eval_quality_score[1h])
    < avg_over_time(agent_eval_quality_score[7d]) * 0.95
  for: 30m
  labels:
    severity: warning
  annotations:
    summary: "エージェント品質スコアが7日平均比5%以上低下"
```

カナリアデプロイとの組み合わせも有効だ。新バージョンへのトラフィックを5%→25%→50%→100%で段階的に移行し、各段階でオンライン評価スコアが閾値内に収まることを確認してから次のステップに進む。エージェントのロールバックトリガーは単純なエラー率だけでなく品質スコアの低下も含める必要がある点が、通常Webサービスのカナリアデプロイとの大きな違いだ。

入力分布の監視も組み合わせると精度が上がる。スコア低下と同時に入力テキストのEmbeddingクラスタリングを確認し、新しいユーザーパターンが増えているかを把握することで、品質劣化の根本原因（モデルの問題なのか入力分布シフトなのか）を素早く切り分けられる。

本番トラフィックのオンライン評価は3つの設計判断で構成される。まずサンプリング方式（ヘッドベース・テールベース・シャドーモード）をシステム特性に合わせて選択する。次にLLM-as-judgeをサンプリングした上位層のみに絞り、ヒューリスティックとの2層構造でコストを制御する。最後に品質スコアを時系列メトリクスとして管理し、カナリアデプロイと連動した自動アラートで劣化を早期検知する。
