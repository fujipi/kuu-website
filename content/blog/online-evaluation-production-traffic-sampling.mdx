本番にデプロイしたAIエージェントが正しく動いているかを確認する手段は、ゴールデンデータセットによるオフライン回帰テストだけではない。オフラインテストはリリース前の品質ゲートとして機能するが、本番環境の入力分布は常に変化し、テストセットが想定しなかったエッジケースが日々発生する。この「本番ドリフト」を検知する仕組みが、オンライン評価——本番トラフィックを継続的に評価基盤に変えるアーキテクチャだ。

本記事は[AIエージェントガバナンス](/ai-governance/)の評価レイヤーを扱い、[LLM-as-a-judgeによるオフライン評価設計](/blog/llm-as-a-judge-agent-evaluation-enterprise/)の後続ステップにあたる。

## オフライン評価の限界と本番ドリフト

> オフラインテストは既知のケースしか検証できず、本番で発生する入力分布のシフトやユーザー行動の変化を捉えることができない。

ゴールデンデータセットへの回帰テストは必要だが十分ではない。本番環境が持つ変数——実ユーザーの予期せぬ入力パターン、モデルプロバイダーのAPI変更、季節変動による入力分布のシフト——はテスト環境では完全に再現できない。本番トラフィックこそが最良のテストケース生成器であり、「本番トラフィックを継続的に拡張するテストデータセットとして扱う」のが2026年の標準的な設計思想だ。

具体的な課題を整理すると以下の3点に集約される:

- **入力分布のシフト**: ユーザーの使い方が変化すると、ゴールデンデータセットの代表性が失われ、オフライン評価スコアが本番品質を反映しなくなる
- **プロバイダーの暗黙的変更**: モデルAPIのバージョン更新で挙動が微妙に変化するが、オフラインテストでは検知のタイムラグが生まれる
- **エージェント連鎖の影響**: マルチエージェント構成では、上流エージェントの変化が下流の品質に波及する。スパン単位での継続監視が必要だ

## サンプリング戦略の設計——2層アーキテクチャ

> 全トラフィックに軽量評価を適用し、5〜10%にLLMジャッジを絞るコスト最適化設計が標準パターンだ。

エンタープライズでのオンライン評価設計の中核は「2層サンプリング」だ。

**第1層：100%カバレッジの軽量評価**

全トラフィックに適用する評価はミリ秒単位で完了するヒューリスティックに限定する。代表的な計測項目は以下の通り:

- **フォーマット検証**: 期待するJSONスキーマへの準拠を即時検証。ツール呼び出しの引数型エラーも検知する
- **レイテンシ計測**: p95/p99レイテンシの逸脱をシグナルとして記録。エージェントのループ異常も早期に捕捉できる
- **出力長の異常検知**: トークン数の急増・急減は、プロンプトの壊れやモデル挙動変化の兆候だ
- **エラーレート**: ツール呼び出し失敗・API例外・タイムアウトをリアルタイム集計する

**第2層：5〜10%のLLMジャッジ評価**

コストの高いLLMジャッジは全トラフィックの5〜10%に絞る（Adaline・FutureAGIが推奨する範囲）。ランダムサンプリングではなく**層化サンプリング**が重要だ。ユーザーセグメント・タスク種別・時間帯ごとに最低サンプル数を保証することで、低頻度だが重要なシナリオが評価から漏れるのを防ぐ。

さらに2〜5%を人間レビューキューに積み上げ、LLMジャッジのキャリブレーションデータとして活用する。本番で検知した失敗ケースは、自動でゴールデンデータセットの候補としてフラグが付与されオフライン回帰テストのフィードバックループが閉じる。

```python
def should_sample_llm_judge(request_metadata: dict) -> bool:
    segment = request_metadata.get("user_segment")
    task_type = request_metadata.get("task_type")
    # 高リスクセグメントは20%、標準は5%
    threshold = 0.20 if segment in HIGH_RISK_SEGMENTS else 0.05
    return random.random() < threshold
```

## シャドーモードとカナリア展開の設計

> シャドーモードは本番入力を受けつつユーザーに出力を届けない「ドライラン」で、リスクゼロで本番挙動を計測する。

新しいエージェントバージョンを本番に投入するとき、Webシステムのカナリアデプロイとは異なるロールアウト設計が必要になる。

**フェーズ1：シャドーモード（ユーザー公開0%）**

シャドーエージェントは本番トラフィックをリアルタイムで受け取り、実際の処理を行うがアクション（メール送信・外部API書き込み・DB更新）は実行しない。シャドーの出力と現行エージェントの出力を比較することで、ユーザーへの影響ゼロで品質ベースラインを確立する。シャドーモードは最低24〜48時間維持し、時間帯・曜日をまたいだ分布のブレを確認することが推奨される。

**フェーズ2：カナリア展開（5% → 20% → 100%）**

シャドーモードで品質スコアが基準を満たしたら、段階的にトラフィックを切り替える。重要なのはロールバックのトリガー条件設計だ。Webシステムのカナリアではエラーレートを指標にするが、AIエージェントでは**品質スコアの閾値割れ**もロールバックトリガーに加える必要がある。たとえばLLMジャッジの「タスク完了率」が3ポイント以上下落した場合や、ポリシー違反率が上昇した場合に自動ロールバックするゲートを設ける。

[エージェントの可観測性とトレース設計](/blog/agent-observability-tracing-instrumentation/)でスパンデータを収集していれば、シャドーモードの比較にスパン粒度の差分を活用でき、問題のあるサブエージェントを特定しやすくなる。

## エンタープライズでの実装パターン

> 複数チームが並行運用するエンタープライズでは、サンプリングレートとコスト配賦をポリシーで集中管理し評価を統制する。

複数の事業部門が独自のAIエージェントを運用する大規模環境では、オンライン評価の設計判断を各チームに委ねると評価基準の不整合が生まれる。以下の要素をポリシーとして集中管理する:

| 管理要素 | 設定内容 | 目的 |
|---|---|---|
| サンプリングレート | 第1層・第2層の割合を部門ポリシーで上限設定 | コスト上限と監査証跡の統一 |
| 評価コスト配賦 | チーム別のLLMジャッジ呼び出し費用を集計 | AI FinOpsの部門レポートに統合 |
| アクセス制御 | 評価ログへのアクセスをIAMスコープで管理 | センシティブな品質データの保護 |
| アラートルーティング | 品質スコア低下時のSlack/PagerDuty通知先 | エスカレーションパスの標準化 |
| 評価器バージョン | LLMジャッジのモデル・ルーブリックバージョンを統一 | チーム間の採点基準の一貫性 |

[LLMゲートウェイ設計](/blog/llm-gateway-routing-rate-limiting/)と評価パイプラインを統合すると、ゲートウェイが採取するリクエスト・レスポンスログを評価ストリームに直接流せる。Langfuse・Helicone等のオブザーバビリティプラットフォームは評価機能を標準搭載し始めており、トレースデータと評価スコアを一元管理できる環境が整いつつある。

オンライン評価基盤の設計・実装をエンタープライズ規模で進めるには、サンプリングポリシー策定・コスト設計・IAM統合・アラート設計まで横断的な実装が必要だ。Kuuの[RDEサービス](https://kuucorp.com/services/rde/)では、評価アーキテクチャの設計から組織横断のガバナンス体制整備まで一貫してサポートしている。

## まとめ

オンライン評価はオフライン回帰テストを代替するものではなく、補完する防衛層だ。両者を組み合わせた評価体制が、本番エージェントの品質を継続的に維持する唯一の方法だ。

実装の優先順位:

1. **第1層の軽量評価を全トラフィックに適用する**: フォーマット検証・レイテンシ・エラーレートから始め、追加コストなしに品質シグナルを得る
2. **LLMジャッジを5〜10%の層化サンプリングで運用する**: ユーザーセグメント・タスク種別ごとに最低サンプル数を保証する
3. **シャドーモードを標準デプロイフローに組み込む**: 新バージョンのリリース前チェックとして義務化し、カナリア展開前に品質スコアの比較を必須とする
4. **本番障害をゴールデンデータセットに自動フィードバックする**: 本番とオフラインの評価ループを閉じて評価基盤を育てる
