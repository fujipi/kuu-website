CIパイプラインのゴールデンデータセットが全パスして本番リリースしたのに、数週間後にエージェントの出力品質が静かに低下していた——これはオフライン評価の構造的な盲点です。モデルのサイレントアップデート、ユーザー行動の変化、外部ツール仕様の微変更は、本番トラフィックにしか現れない品質劣化要因です。

エンタープライズでAIエージェントを継続運用するには、デプロイ後も本番トラフィックを評価し続ける**オンライン評価**パイプラインが必要です。本記事は[AIエージェントガバナンス](/ai-governance/)のピラーコンテンツに連動し、本番評価の設計パターンを具体的に解説します。

## オフライン評価の限界とオンライン評価が必要な理由

> オフライン評価は既知の失敗パターンを検出するが、本番でのモデルドリフトとデータ分布変化は本番サンプリングでしか捕捉できない。

[LLM-as-a-judge評価基盤](/blog/llm-as-a-judge-agent-evaluation-enterprise/)で解説したゴールデンデータセットと回帰テストは、「既知の品質基準に照らした検証」に特化しています。CIパイプラインのゲートとして機能しますが、デプロイ後の本番環境で発生する変化には対応できません。

**オフライン評価では捕捉できないリスク：**

- **モデルドリフト**: ベースLLMがサイレントアップデートされた場合、ゴールデンデータセットでは変化が出なくても本番挙動が変化する
- **データ分布シフト**: 実ユーザーのリクエスト分布がテストケースから乖離していく（季節変動、製品変更等）
- **ツール依存の変化**: 外部APIやデータベーススキーマの微変更がエージェントの実行経路に影響する
- **未知の失敗モード**: 本番にしか現れない入力の組み合わせパターン

「本番でしか起きない問題は本番でしか見えない」——この前提からオンライン評価の設計が始まります。

## 本番トラフィックサンプリング戦略

> 全リクエストの5〜10%にLLMジャッジ採点を走らせ、残りは軽量ヒューリスティックで監視するのが費用対効果の高い設計です。

本番評価の課題はLLMジャッジ採点のコストと評価カバレッジのトレードオフです。Adalineの2026年ガイドでは「高コストなLLMジャッジは本番リクエストの5〜10%に適用し、残りは決定論的チェックと軽量メトリクスで監視する」構成が標準パターンとして示されています。

**二層サンプリング設計：**

| 層 | 対象 | 手法 | コスト |
|---|---|---|---|
| 全トラフィック層 | 100%のリクエスト | レイテンシ・エラーレート・トークン数の決定論的チェック | 低 |
| 評価サンプリング層 | 5〜10%（リスクベース） | LLMジャッジによる質的採点 | 中〜高 |

ランダムサンプリングより**リスクベースサンプリング**が有効です。エラー率の高いユーザーセグメント、新機能パス、コンプライアンス要件のある業務フロー（金融・医療・法務）を優先的にサンプリングします。

```python
def should_evaluate(trace: AgentTrace) -> bool:
    # 高リスクパスは必ず評価
    if trace.workflow_type in HIGH_RISK_WORKFLOWS:
        return True
    # エラーまたは再試行が発生したトレースは必ず評価
    if trace.has_tool_errors or trace.was_retried:
        return True
    # それ以外は5%のランダムサンプリング
    return random.random() < 0.05
```

統計的有意性の観点では、品質スコアの±2%変化を95%信頼区間で検出するには週あたり約400サンプルが目安です（ベースラインスコア80%・検定力80%の場合）。

## シャドウモードとカナリアデプロイによる段階的リリース

> シャドウモードはユーザーに影響ゼロで候補モデルを本番評価し、精度が閾値を超えたら段階的にカナリア展開する設計です。

新モデルバージョンや大規模プロンプト改修のリリースには、シャドウモードによる事前検証が不可欠です。TianPanの技術ガイドが示すシャドウデプロイの構成は次のとおりです：

1. **本番ルーター**: 全リクエストを現行モデルと候補モデルの両方に並行送信する
2. **候補モデル**: レスポンスをユーザーには返さず、ロギングシステムに書き込む
3. **比較評価**: LLMジャッジが現行vs候補のレスポンスをスコアリングし、差分を集計する
4. **リリース判定**: 候補モデルのスコアが現行を上回り、かつ閾値（例：95%以上）に達したらカナリア展開を開始する

カナリア展開では1% → 5% → 20% → 100%の段階的トラフィック移行と、各段階でのSLOチェックを組み合わせます。金融・コンプライアンス系の業務エージェントでは、各フェーズに最低24時間の観測ウィンドウを設けることを推奨します。

シャドウモードの副次的な価値として、候補モデルのレスポンスをゴールデンデータセットの新規ケース候補として収集できる点があります。本番で初めて遭遇した入力パターンを半自動的にテストスイートに組み込めます。

## 品質ドリフト検出とアラート設計

> エージェントの品質ドリフトは緩やかに進行するため、7日移動平均と制御限界を組み合わせた統計的検出が有効です。

本番エージェントの品質低下は多くの場合、単一イベントではなく数週間かけて緩やかに進行します。MLflowのAI監視フレームワークが推奨するのは、7日移動平均の品質スコアに対する制御限界（UCL/LCL）の設定です。

**アラートシグナルの分類：**

| シグナル | 検出方法 | アクション |
|---|---|---|
| スコア急落（5%以上の即時低下） | 静的閾値 | PagerDuty即時通知・ロールバック検討 |
| 緩やかなドリフト（週次2%低下が3週継続） | 移動平均×制御限界 | Slack通知・原因調査 |
| 特定フロー/セグメントの外れ値 | Zスコアによる異常検出 | 対象ルートの深掘りサンプリング |

[エージェントの可観測性設計](/blog/agent-observability-tracing-instrumentation/)で収集したスパンデータを品質評価のインプットに活用できます。ツール呼び出し数の増加・再試行率の上昇・レイテンシ外れ値を品質劣化の先行指標として設定すると、LLMジャッジ採点コストを抑えながら早期検出が可能です。

品質ドリフト検出から得たフィードバックをゴールデンデータセットに昇格させるループも重要です。本番で発見された新規失敗パターンをオフラインテストスイートに組み込むことで、CI回帰テストの網羅性が継続的に向上します。

大規模エンタープライズでオンライン評価パイプラインを設計・運用するには、サンプリング基盤・LLMジャッジ統合・ドリフト検出・ガバナンス報告を横断したアーキテクチャ設計が必要です。Kuuの[RDEサービス](https://kuucorp.com/services/rde/)では、評価基盤の設計から本番導入まで一貫して支援しています。

## まとめ

オンライン評価はオフライン評価の代替ではなく補完です。ゴールデンデータセットがデプロイのゲートを守り、オンライン評価が本番品質を継続監視する——この二層構造がエンタープライズエージェントの品質管理の基本形です。

実装の優先順位：

1. **全トラフィック層の軽量監視**: エラーレート・レイテンシ・トークン数の決定論的チェックを先に整備する
2. **5〜10%のLLMジャッジサンプリング**: リスクベースサンプリングで高優先ルートを優先評価する
3. **シャドウモードの整備**: 大規模変更のリリース前に並行評価パイプラインを構築する
4. **ドリフト検出アラート**: 7日移動平均×制御限界で緩やかな品質低下を早期に検出する

品質ドリフトに気づかないまま運用を続けることは、エージェントによる誤判断の自動化を続けることと同義です。エンタープライズレベルの評価基盤設計は[Kuuの企業向けRDEサービス](https://kuucorp.com/services/rde/)にご相談ください。
